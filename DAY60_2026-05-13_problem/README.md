Day 60 — Bank Account Balance Reconstruction from Event Ledger
Problem Overview

Modern banking systems generate massive streams of transactional events every second.
However, reconstructing the actual account balance is rarely straightforward.

In real-world financial systems:

transactions can fail
retries may create duplicate records
chargebacks may later get reversed
refunds can offset earlier losses
correction events may arrive days later
multiple events can reference the same financial operation

This challenge focuses on rebuilding the true daily closing balance for every account using a raw event ledger.

The goal is to simulate how financial systems reconstruct balances from imperfect transactional data.

Business Objective

Build a reliable ledger reconstruction pipeline that:

removes duplicate financial events
ignores invalid transactions
correctly processes reversal events
rebuilds historical balances
fills missing calendar dates
generates accurate daily closing balances

This closely reflects real-world banking, fintech, and payment reconciliation systems.

Table Description
account_events

Contains raw transactional events generated by the banking system.

Each row represents an event that may or may not affect the final balance.

Event Types

The dataset includes multiple types of financial activity.

deposit

Money credited into the account.

Balance impact:

+amount
withdrawal

Money debited from the account.

Balance impact:

-amount
fee

Charges deducted by the bank.

Balance impact:

-amount
refund

Money returned back to the account.

Balance impact:

+amount
chargeback

Forced deduction initiated due to disputes or payment reversals.

Balance impact:

-amount
reversal

A corrective event that reverses the impact of an earlier transaction.

The reversal references another transaction using:

reference_id

The reversal event itself does not use its own amount value.

Instead, its impact depends on the original referenced transaction.

duplicate

Represents duplicated ingestion or retry records.

These events should never affect balance calculations.

Rules to Consider Before Solving
1. Only Successful Events Are Valid

Only events with:

status = SUCCESS

should participate in balance calculations.

All failed events must be ignored completely.

2. Ignore Duplicate Financial Events

Multiple rows may represent the same financial operation.

If multiple successful financial events share the same:

reference_id

only the earliest valid event should be considered.

Later duplicates must be ignored.

This applies to:

deposits
withdrawals
refunds
fees
chargebacks
3. Duplicate Event Type Must Be Ignored

Rows where:

event_type = 'duplicate'

must never affect balances.

These rows simulate ingestion retries or duplicated message processing.

4. Reversal Logic

A reversal event cancels the effect of an earlier transaction.

The reversal references another transaction using:

reference_id

The reversal amount column should not be used directly.

Instead:

reversing a deposit removes money
reversing a withdrawal restores money
reversing a fee restores money
reversing a refund removes money
reversing a chargeback restores money
Reversal Impact Rules
Reversing a Deposit

Original:

+500

Reversal impact:

-500
Reversing a Withdrawal

Original:

-200

Reversal impact:

+200
Reversing a Fee

Original:

-25

Reversal impact:

+25
Reversing a Refund

Original:

+80

Reversal impact:

-80
Reversing a Chargeback

Original:

-120

Reversal impact:

+120
Multiple Reversal Handling

A transaction may accidentally receive multiple reversal rows.

Only the first valid reversal should be applied.

All additional reversals for the same referenced transaction must be ignored.

Invalid Reversal Handling

If a reversal references:

a failed transaction
an ignored duplicate transaction
a non-existing reference_id

then the reversal should contribute:

0

to the balance.

Daily Balance Reconstruction

The final output must generate one row per:

account_id + date

between the first and last activity date of that account.

Missing Date Handling

Some days may contain no valid financial activity.

These dates must still appear in the output.

For inactive dates:

daily_net_amount = 0

The closing balance should carry forward from the previous day.

Expected Output

The final result should contain:

account_id

Unique account identifier.

balance_date

Calendar date for the reconstructed balance.

daily_net_amount

Net financial impact for that specific date after:

deduplication
reversal handling
invalid event removal
closing_balance

Running closing balance after applying all valid daily activity.

Edge Cases Included in Dataset

The dataset intentionally includes multiple real-world complications.

Duplicate Deposits

Same transaction inserted multiple times.

Failed Transactions

Events that should not affect balances.

Duplicate Reversals

Same transaction reversed more than once.

Reversal of Negative Transactions

Withdrawals and chargebacks being reversed.

Missing Calendar Dates

Accounts with inactive days.

Invalid Reference IDs

Reversals pointing to missing transactions.

Mixed Financial Corrections

Transactions being offset days later.
